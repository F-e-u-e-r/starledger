# P3.2 executor runbook

How to run a StarLedger classification executor. The executor is the only
model-calling component and runs **outside** this repository; CI never calls a
model. Enable exactly **one** scheduled executor at a time.

## Shared contract

- Prompt: `prompts/classify-agent-v1.md` (version `classify-v1`).
- The executor may modify only `ai-annotations.json` and `ai-annotations-meta.json`.
- Temporary `.ai-runs/manifest.json` and `.ai-runs/candidates.json` are git-ignored
  and must never be committed.
- No direct push to `main`, no state-branch push, no auto-merge.
- The `starledger-ai-state` branch is maintained by the trusted `ai-state.yml`
  workflow; the executor only READS it through `plan` (never `--save-state`).

## One-time provenance-gate bootstrap

Before enabling either executor, merge the reviewed P3.3 implementation PR with
no AI artifact files. This is the one unavoidable `pull_request_target`
bootstrap: the new provenance workflow cannot run until its trusted workflow and
CLI are already on `main`. Then open a source-only PR, confirm the green
`verify-ai-provenance` no-op check appears, and add it alongside
`verify-agent-artifacts` as a required `main` status check. Only then enable one
executor to open `claude/*` or `codex/*` artifact PRs.

Create and commit `config/ai.yaml` from `config/ai.example.yaml` before the first
executor run. It contains no secret. Set `ai.enabled: true`, select exactly one
`executor_kind`, and start with a small valid budget. Without that explicit opt-in
the planner writes an empty manifest and the provenance gate rejects any artifact
PR.

## Deterministic command sequence

Run with a GitHub token in `STAR_SYNC_TOKEN` or `GITHUB_TOKEN` (README discovery):

```text
pnpm classifier plan --out .ai-runs/manifest.json --current ai-annotations.json
#   the agent reads .ai-runs/manifest.json, classifies each job, and writes
#   .ai-runs/candidates.json per prompts/classify-agent-v1.md
pnpm classifier validate-candidates --manifest .ai-runs/manifest.json \
    --candidates .ai-runs/candidates.json
pnpm classifier apply --manifest .ai-runs/manifest.json \
    --candidates .ai-runs/candidates.json \
    --generated-at <iso> --current ai-annotations.json --out-dir .
pnpm classifier verify-artifacts --annotations ai-annotations.json \
    --meta ai-annotations-meta.json
#   commit ONLY the two artifacts on an executor branch and open a PR
```

`apply` reconciles candidates against the manifest: it applies valid candidates,
reports jobs still pending retry, and hard-fails on any rejected (stale, wrong
executor, or smuggled) candidate.

## Claude Routine (primary)

- Cloud-hosted, scheduled on the Anthropic platform — not a GitHub Action.
- Restricted `claude/` branch policy; no unrestricted push, no auto-merge.
- Minimal connectors / network; treats all repository material as untrusted data.
- Opens a `claude/*` PR with only the artifact pair; a human reviews and merges.

## Codex App Automation (fallback)

- Runs locally via ChatGPT auth in a **new git worktree**, so it never disturbs an
  active working tree.
- Same prompt and same CLI sequence; opens a local branch or `codex/*` PR.
- Not scheduled concurrently with Claude Routine.

## Meta-rebase (ROAD-A, optional, model-free)

When the daily sync updates `stars.json` while an executor PR is still open, its
`ai-annotations-meta.json.dataset_sha256` no longer matches the current base and
`verify-ai-provenance` (PROV-5) rejects the PR. Instead of re-running the model,
re-stamp the meta onto the current base with `rebaseAiAnnotationsMeta`
(`@starred/classifier`): it validates the head artifact pair (schema, canonical
form, hash, count, taxonomy), re-runs the per-annotation provenance checks against
the current base with the dataset pointer forced to current, and emits a meta with
ONLY `dataset_sha256` changed (annotation bytes and `generated_at` preserved). It
does NOT re-check PROV-5 — it establishes the invariant by re-stamping after those
checks pass; the live `verify-ai-provenance` gate re-runs the REAL PROV-5 at merge
as the final authority (and rejects a metadata-only change, which the helper also
refuses).

Run it manually (needs a token for README discovery, like `plan`):

```text
# report-only (verify the head is still valid against the current base):
# pass --base-annotations <base>/ai-annotations.json, or --cold-start if the base
# has no annotations yet (the first AI PR).
pnpm classifier meta-rebase --stars stars.json --meta dataset-meta.json \
    --head-annotations ai-annotations.json --head-meta ai-annotations-meta.json \
    --base-annotations <base>/ai-annotations.json --dry-run
# re-stamp: write the updated pair to <out> (commit ONLY those two files):
pnpm classifier meta-rebase --stars stars.json --meta dataset-meta.json \
    --head-annotations ai-annotations.json --head-meta ai-annotations-meta.json \
    --base-annotations <base>/ai-annotations.json --out-dir .
```

This is a MANUAL, model-free helper. It is NOT wired into any workflow, required
check, or ruleset bypass, it does not change PROV-5, and a re-stamped PR still
passes through BOTH the structural and provenance gates and human review like any
other. See `docs/adr/ADR-002-meta-rebase.md` for the threat model.

## What still gates the PR

- The structural gate (`.github/workflows/ai-agent-pr.yml`) accepts only the
  complete artifact pair from an approved, same-repository executor branch.
- The P3.3 provenance gate additionally recomputes current jobs/fingerprints and
  rejects stale or invented annotations. Keep it required alongside the structural
  gate, and do not auto-merge executor PRs.
